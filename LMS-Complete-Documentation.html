5. Frontend Pages → API Calls

Every page & every API call
5.1 Auth Pages
LoginPage.jsx — /login
POST /api/auth/login
Body: { email, password }
Returns: { accessToken, user: {id, fullName, email, role, profilePicture} }
On success: store accessToken in memory, redirect to /dashboard
POST /api/auth/google (Google button click)
Body: { token: googleIdToken }
Returns same shape as login
RegisterPage.jsx — /register
POST /api/auth/register
Body: { fullName, email, contactNumber, password }
Returns: { accessToken, user }
On success: auto-login, redirect to /dashboard
5.2 Dashboard Pages
DashboardPage.jsx — /dashboard

Reads role from AuthContext (no API call needed — user already in context from login). Renders AdminDashboard, TeacherDashboard, or StudentDashboard based on role.

AdminDashboard
Navigation cards only — links to /admin/users, /admin/courses, /tests, /admin/mcq-reports. No API calls on mount.
TeacherDashboard
Navigation cards only — links to /tests, /teacher/mcq-reports. No API calls on mount.
StudentDashboard
Navigation cards — links to /student/courses, /student/tests, /student/mcq-reports. No API calls on mount.
⚠️ IssueAll three dashboard components show static navigation cards only. Zero real-time data. This means the dashboard gives no actual insight — a missed opportunity to show stats like tests completed today, recent scores, pending reports, etc.
5.3 Test Management Pages (Admin/Teacher)
TestListPage.jsx — /tests
GET /api/tests — on mount, fetches ALL tests (no server-side pagination)
GET /api/question-banks — for QB filter dropdown
GET /api/courses — for Course filter dropdown
🔴 Performance IssueLoads ALL tests at once then filters client-side. With 1000+ tests this becomes very slow. Needs server-side pagination + filtering.
TestFormPage.jsx — /tests/create or /tests/:id/edit
GET /api/tests/:id (edit mode only) — pre-fill form
POST /api/tests (create) — Body: { title, description, passingScore, difficultyLevel, instructions }
PUT /api/tests/:id (edit) — same body
On success: navigate to /tests/:id (TestDetailPage)
TestDetailPage.jsx — /tests/:id
GET /api/tests/:id — on mount (returns test + all populated MCQs)
GET /api/mcqs/test/:testId — separate call for MCQ list display
PUT /api/tests/:id/publish — Publish button click
🔴 Duplicate API CallgetTest() already populates MCQs (test.mcqs[]), yet the page also calls getMCQsForTest() separately. This is two DB round-trips fetching the same data. Only one call needed.
5.4 Student Test-Taking Flow (the most important flow)
TestHistoryPage.jsx — /student/tests
GET /api/user-tests/history — on mount; returns all attempts with nested test → QB → Course populate
Client-side: filters by status, subject, QB, search text. All filtering local.
TestStartPage.jsx — /student/tests/:testId
GET /api/tests/:testId — test info (title, description, totalQuestions, passingScore, difficultyLevel)
GET /api/user-tests/history — check for existing in-progress attempt for this test
POST /api/user-tests/start (Start Test button) — Body: { testId, mode: 'tutor'|'timer' }
Returns: full UserTestAttempt with populated MCQs
On success: navigate to /student/tests/:testId/play?attemptId=xxx&duration=yyy (seconds for timer mode)
🔴 InefficiencyTestStartPage fetches the full user history (ALL attempts, all tests) just to check if one test has an in-progress attempt. Should call a dedicated endpoint: GET /api/user-tests/active?testId=xxx
TestPlayerPage.jsx — /student/tests/:testId/play
Reads ?attemptId and ?duration from URL params.
GET /api/user-tests/:attemptId — on mount (get full attempt with MCQ details)
PUT /api/user-tests/:attemptId/question/:index — on every answer select
Body: { selectedOption: 'A'|'B'|'C'|'D'|'E' }
Returns: { isCorrect, correctOption, explanation, attempt }
PUT /api/user-tests/:attemptId/report/:index — Report button
Body: { reportReason }
PUT /api/user-tests/:attemptId/save/:index — Save button
Body: { notes, taggedCategories }
PUT /api/user-tests/:attemptId/mark/:index — Mark for Review toggle
PUT /api/user-tests/:attemptId/complete — Submit Test button
Body: { totalTimeSpent }
On success: navigate to /student/tests/:testId/result/:attemptId
🔴 Critical Performance IssueEvery single answer selection fires an API call immediately. For 200 questions in a mock exam with 1000 students = potentially 200,000 API calls per exam session. Should debounce or batch these into periodic saves (every 30s) + on navigation. Only save immediately for tutor mode feedback.
TestResultPage.jsx — /student/tests/:testId/result/:attemptId
GET /api/user-tests/:attemptId — load attempt with score details
Displays: score %, correct/wrong counts, pass/fail badge, links to review
TestAttemptReviewPage.jsx — /student/tests/:testId/review/:attemptId
GET /api/user-tests/:attemptId — full attempt with all MCQ details
Displays: each question with student's answer highlighted, correct answer, explanation
