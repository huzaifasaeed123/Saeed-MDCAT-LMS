Mode 1 — Multi-Device, Long-Lived Session

Current implementation + admin-configurable duration

How It Works Today

Refresh token is a signed JWT verified only by its cryptographic signature — no DB storage. Works across every device the user logs into. A student can have the app open on phone, laptop, and tablet simultaneously with zero conflict.

What Adding Admin-Controlled Duration Requires

Currently the session duration is hardcoded as '547d' in User.js. To let admin change this:

1
Add a Settings model in DB Store sessionDurationDays (e.g. 365, 547, 730) alongside other global settings.
2
Read setting at login time When issuing the refresh token, do 1 extra DB read to get current sessionDurationDays, then pass it as expiresIn to jwt.sign().
3
Match cookie expiry to JWT expiry setRefreshCookie must use the same duration for the cookie's expires field, not the hardcoded 547 days.
4
Admin changes the setting Only affects NEW logins. Existing tokens keep their original expiry — JWT expiry is baked in at signing time and cannot be changed retroactively. This is the correct behaviour (existing users are not abruptly logged out).
Important: If admin reduces duration from 2 years to 30 days, existing logged-in users are NOT affected immediately. Their current refresh tokens keep working until their original expiry. Only new logins will get the shorter duration. This is by design — same as how banks handle session policy changes.

Mode 2 — Single Session (Last Device Wins)

Only one active session at a time; new login kicks old device

How It Works

When a user logs in, a new sessionVersion number is incremented in the DB and embedded in both the access token and refresh token payload. When the old device tries to use its refresh token, the version in the token no longer matches the DB — it gets a 401 and is logged out.

Why version number, not storing the raw token? Storing the raw refresh token in DB brings back rotation and race conditions. A simple integer version is smaller, faster to compare, and achieves the same goal.

Login Flow

1
User logs in from Device B Backend does: User.findByIdAndUpdate(id, { $inc: { sessionVersion: 1 } })
2
Embed version in both tokens Access token payload: { id, role, fullName, sv: 3 }. Refresh token payload: { id, sv: 3 }
3
Device A tries to refresh (1 hour later) Backend verifies JWT → decodes sv: 2 → DB has sv: 3 → mismatch → 401 → Device A is logged out
4
Device A's access token still works for up to 1 hour This is the accepted window. You could check sv on every request but that kills the "no DB per request" benefit. 1h window is acceptable for most use cases.

DB Schema Change

Two-Tab Problem with Single Session Mode: If a user has two tabs open, both tabs have the same session version and the same refresh token. Tab A refreshes → version increments to 4. Tab B also had version 3 in its token → Tab B gets kicked out after 1h. This is expected and correct behaviour for "single session" mode. The user chose this restriction.

Pros

  • No persistent session storage — just one integer per user
  • Old device is cleanly kicked within 1 hour (when its access token expires and it tries to refresh)
  • Very low DB overhead — same as Mode 1 (zero per request, one per refresh)
  • Admin can force logout all sessions by bumping the version manually

Cons

  • Old device keeps working for up to 1 hour (access token window)
  • Multiple tabs in same browser will eventually conflict (user chose single-session, so this is correct)
  • No "which device" visibility — you know someone is logged in somewhere, but not where

Mode 3 — Device-Limited Sessions (Max N Devices)

Track every active device; block or evict when limit is exceeded

How It Works

Each login creates a session record in the DB with a unique session ID, device info, and last-active timestamp. The refresh token carries the session ID. On every refresh, that specific session record is updated. When the device count exceeds the admin-set limit, the new login is either blocked or the oldest inactive session is evicted.

Login Flow

1
User logs in from Device C Backend counts active sessions for this user. If count < maxDevices → proceed. If count ≥ maxDevices → evict oldest OR block login.
2
Create session record in DB Store: { userId, sessionId (uuid), deviceName, userAgent, ip, createdAt, lastActiveAt }
3
Embed sessionId in refresh token Refresh token payload: { id, sid: 'uuid-here' }. Cookie is set as usual.
4
Every token refresh Backend verifies JWT, extracts sid, checks it exists in DB, updates lastActiveAt. This proves the session is still valid.
5
Admin revokes a device Admin deletes the session record for that sid. Next time that device tries to refresh → session not found → 401 → logged out.

DB Schema — New Sessions Collection

// New "sessions" collection (separate from User) { _id: ObjectId, userId: { type: ObjectId, ref: 'User', index: true }, sessionId: { type: String, unique: true }, // UUID v4 deviceName: String, // 'Chrome on Windows', 'Safari on iPhone' ip: String, createdAt: Date, lastActiveAt: Date, expiresAt: Date // TTL index — auto-delete expired sessions } // MongoDB TTL index — sessions auto-delete after expiry (no manual cleanup needed) SessionSchema.index({ expiresAt: 1 }, { expireAfterSeconds: 0 }); SessionSchema.index({ userId: 1 });

Device Detection (Frontend)

On login, the frontend sends basic device info in the request body:

// Send with login request { email: '...', password: '...', deviceName: `${navigator.userAgent}` // parse on backend } // Backend uses 'ua-parser-js' npm package to extract 'Chrome on Windows 11'

Admin Capabilities with This Mode

  • See list of all active devices for any user (device name, IP, last active)
  • Revoke any specific device immediately
  • Set max devices per role (students: 3, teachers: 5)
  • Set behaviour when limit is hit: block new login OR evict oldest device
  • Force logout all devices for a user (delete all sessions for userId)

Pros

  • Full device visibility and control for admin
  • Users can see their own active devices and revoke them
  • Strongest security model
  • Natural fit for enterprise/subscription SaaS

Cons

  • Extra DB collection to maintain
  • Every token refresh = 1 DB read + 1 DB write (lastActiveAt update)
  • Most implementation work of the three modes
  • Device fingerprinting is imperfect (incognito = different "device")

Combined SaaS Architecture — All 3 Modes in One System

Admin switches between modes via settings; code handles all three paths

How the Admin Switches Modes

One field in the Settings document controls which mode is active: sessionMode: 'multi' | 'single' | 'device-limited'. The backend reads this on login and on refresh to decide which path to take.

Server-side caching of settings is essential. If every API refresh had to read the Settings document to know which mode is active, that's 1 extra DB read per refresh per user. Instead, cache settings in application memory and refresh it every 5 minutes. On a server with 1000 concurrent users, this turns 1000 DB reads/refresh-cycle into 1.

Code Architecture — How to Manage All Three

The cleanest approach is a session strategy pattern. One file per mode, a factory that selects the right one based on cached settings:

// backend/services/session/ │ ├── multiSessionStrategy.js // Mode 1: just jwt.sign(), no DB touch ├── singleSessionStrategy.js // Mode 2: increment sessionVersion ├── deviceLimitedStrategy.js // Mode 3: sessions collection CRUD └── sessionFactory.js // reads cached settings, returns right strategy // authController.js calls: const strategy = await getSessionStrategy(); // from cache, never hits DB per-request const tokens = await strategy.createSession(user, req); const valid = await strategy.validateRefresh(decoded, req); await strategy.destroySession(decoded, res);

Settings Cache Pattern

// backend/services/settingsCache.js let _cache = null; let _lastFetched = 0; const TTL = 5 * 60 * 1000; // 5 minutes async function getSettings() { if (_cache && Date.now() - _lastFetched < TTL) return _cache; _cache = await Settings.findOne(); _lastFetched = Date.now(); return _cache; } // 1 DB read per 5 minutes total, regardless of how many users are active

What Happens on Mode Switch

  • Multi → Single: Existing multi-device users keep working until their next refresh. At that point, the new mode is enforced and old sessions start getting invalidated. Gradual rollout automatically.
  • Single → Device-Limited: The sessionVersion field becomes unused. New logins start creating session records. Old single-mode tokens (without sid) would fail device-limited validation — admin may need to force re-login all users during the migration.
  • Any → Multi: Immediately relaxes restrictions. All existing tokens that are JWT-valid work again.
Recommendation for SaaS: Build Mode 1 now (it's done). Build Mode 2 next (one DB field, 2 days of work). Build Mode 3 only when a specific customer asks for device management features — it's significant work and most LMS users don't need it.

Recommendation for Your SaaS LMS

Build Now (Already Done)

Mode 1 with admin duration control. Add sessionDurationDays to Settings model. This covers 90% of your users perfectly. Zero extra burden.

Build in Sprint 2 (~2 days)

Mode 2 — single session. Add sessionVersion to User model and a version check in the refresh endpoint. Gives you the "only one session" selling feature for strict-access plans.

Build When Customer Asks

Mode 3 — device-limited. This is the most work (new collection, admin device management UI, device fingerprinting). Build it for your first enterprise customer who needs it — don't build it speculatively.
