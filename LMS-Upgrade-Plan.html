Ask a Question — Q&A Forum System

High Priority
🔵 Core Q&A Forum (Per-MCQ & General)
A student can ask a question on any MCQ page ("I don't understand why option B is wrong") OR post a general question to a topic/subject forum. Teachers and admins can answer. The answer can be marked Public (visible to all students) or Private (only to the asker). Students can up-vote helpful answers.

What to Build

  • New Question model: title, body (rich text), mcqId (optional), subject/topic tags, askedBy, isPublic, status (open/answered/closed)
  • New Answer model: questionId, body, answeredBy, isAccepted, upvotes[]
  • API: POST /questions, GET /questions (with filters), POST /questions/:id/answers
  • Frontend: Q&A page per MCQ, global forum page, rich text answer editor
  • Notification: alert teacher/admin when new question is posted

Public vs Private Toggle

  • Student asks question — by default private
  • Teacher answers → can toggle "Make Public" before saving
  • Public Q&As become a searchable knowledge base
  • Admin can bulk-approve public answers
  • Students can search solved questions before asking new ones (reduces duplicates)
Complexity
Medium
Backend: 2 new models Frontend: 3–4 new pages Effort: 1–2 weeks
🟡 Coin Economy — Spend to Ask, Earn to Answer
Each student gets a starting balance of coins. Asking a question costs coins. Answering another student's question (peer help) earns coins. The system self-regulates: students only ask serious questions, and are motivated to help peers.

Coin Rules (Suggested)

  • New account: 20 free coins
  • Ask a question: −5 coins
  • Your answer is accepted: +10 coins
  • Your answer gets 5 up-votes: +3 bonus coins
  • Daily login streak: +1 coin/day
  • Complete a test (score ≥ 70%): +3 coins
  • Buy coins: optional paid pack (monetization)

What to Build

  • Add coins (int) and coinTransactions[] to User model (or separate Wallet model)
  • Transaction log: amount, reason, relatedId, createdAt
  • API: GET /wallet/balance, GET /wallet/history
  • Frontend: Coin balance widget in navbar, transaction history page
  • Gate: "Ask Question" button disabled if coins < 5 (with "Earn More" link)
Complexity
Medium
Backend: Wallet model + middleware Frontend: Coin badge, transaction page Effort: 1 week
🟢 Extra Ideas to Make Q&A Unique
Additional features that will make the Q&A system feel premium and MDCAT-specific.

Smart Duplicate Detection

  • Before student submits question, search existing Q&As for similar keywords

Expert Badge on Answers

Competition is the #1 driver of daily platform usage for MDCAT students. Below are 5 different leaderboards each measuring a different dimension, so every type of student has a chance to shine.

# Leaderboard Name What It Measures Reset Cycle Why It's Unique
1 🥇 Overall Score Board Total score % averaged across all completed tests All-time + Monthly Classic — rewards accuracy
2 ⚡ Speed & Accuracy Board Correct answers per minute (score ÷ time spent) Weekly Rewards fast + accurate students (MDCAT exam is timed)
3 📈 Most Improved Board Score improvement % over last 30 days (last 5 tests vs previous 5) Monthly Encourages struggling students — they can win too
4 🔥 Streak Board Longest consecutive daily practice streak All-time + Weekly Daily habit-forming — breaks = reset, so students keep coming back

📊 How to Track Performance

  • Update stats after every test attempt completion (hook or post-save middleware).
  • Leaderboard API: aggregate query sorted by the relevant metric, paginated (top 100), filtered by optional course/subject.
  • Cache top-50 in Redis or a simple DB field refreshed every 15 min to avoid heavy aggregations on each page load.

🔬 Weak Topic Detection — How to Build It
This is the most valuable analytical feature. Here is the exact technical approach:

Data Source (already available)

  • UserTestAttempt.questionAttempts[].isCorrect — correctness per Q
  • MCQ.qbSubjectId, qbChapterId, qbTopicId — classification
  • MCQ.subject, unit, topic — direct string tags
  • Join these to compute per-topic accuracy for each student

Aggregation Pipeline (MongoDB)

  • $lookup: attempts → MCQs (join on mcqId)
  • $group by topic: count total, count correct
  • $project: accuracy = correct/total × 100
  • $sort by accuracy ASC → weak topics on top
  • Run on demand or cache daily per student in UserStats
Complexity
High (but High Value)
MongoDB $lookup + $group aggregation New /analytics/student/:id endpoint Charts: recharts or chart.js Effort: 2–3 weeks full implementation

⚡ Practical Advice on Chat

!
Chat is the most complex feature on this list. It needs WebSocket server (Socket.IO), message persistence, room management, and content moderation. Consider using a third-party service like Sendbird, Stream Chat, or Pusher to save 3–4 weeks of work. They have React SDKs that drop in easily.
!
Minimum viable version: Start with a simple "Discussion Thread" per course (like a comment feed, no real-time). Students post messages, teachers reply. No WebSocket needed. Takes 1 week. Then upgrade to real-time later.

High Priority — Direct MDCAT Value

Every MCQ a student ever answered wrong, saved, or flagged for revision is stored in their personal notebook. The student can filter, add notes, and re-attempt them at any time. Your SavedQuestion model already exists — expand it!

  • Spaced Repetition Tag: system marks MCQ for review in 1/3/7 days
  • Export to PDF: student downloads their weak MCQs as a PDF for offline study
  • Share with friend: shareable link to a specific MCQ (read-only)
Complexity

🔁 Spaced Repetition — Never Forget Again
Based on the SM-2 algorithm (used by Anki), wrongly answered MCQs get scheduled for review at increasing intervals: 1 day → 3 days → 7 days → 14 days → 30 days. Once correct, interval increases; if wrong again, resets to 1 day.

How to Build

  • Add to SavedQuestion: nextReviewDate, interval (days), easeFactor, repetitions
  • After each attempt: run SM-2 update on the MCQ record
  • Daily notification: "You have 12 MCQs due for review today"
  • Review mode: special test player showing only due cards

Why This is Unique for MDCAT

  • No other MDCAT platform in Pakistan offers spaced repetition
  • Students will study smarter — remember more with less time
  • Directly maps to MDCAT exam preparation strategy
  • Strong retention means students stay on platform until exam day
Complexity
High (algorithm + scheduling)
SM-2 Algorithm Scheduled review API endpoint Effort: 2 weeks

🤖 AI-Powered Weak Area Diagnosis
Use the OpenAI API (or any LLM) to generate a personalised study recommendation paragraph for each student every week: "Based on your last 7 tests, you are weak in Mitosis under Cell Biology. Focus on the differences between metaphase stages. Recommended: take Test #47."

What to Build

  • Weekly scheduled job (cron): compute each student's weak topics
  • Build a prompt from their stats: "Student got 34% in Mitosis, 29% in Meiosis, 91% in DNA Replication..."
  • Call GPT-4o API → get recommendation paragraph
  • Store in UserStats.weeklyRecommendation
  • Display in student dashboard as "Your Weekly AI Diagnosis"

Effort & Cost

  • Backend: cron job + OpenAI call (~$0.001 per student per week)
  • For 1000 students: ~$1/week → very affordable
  • Frontend: styled "AI Insight" card
  • Effort: 1–2 weeks
  • No MDCAT platform in Pakistan has this — massive USP
